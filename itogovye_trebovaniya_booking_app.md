# Итоговые требования к приложению бронирования игровых столов

## 0. Назначение документа

Документ фиксирует итоговую версию требований к приложению для бронирования игровых столов в клубе настольных игр и варгеймов.

Проект преследует две цели:

1. Создать работающее приложение для реального использования одним клубом.
2. Использовать проект как портфолио-кейс для вакансий уровня **AI-assisted developer**.

Проект должен публично демонстрировать процесс разработки с помощью ИИ и Codex: prompt log, dev diary, README, архитектурные решения, тесты, PR-процесс и документацию.

---

# 1. Цель и формат проекта

## 1.1. Формат

Приложение создаётся для **одного конкретного клуба настольных игр и варгеймов**.

На данном этапе проект не является SaaS-платформой для множества клубов. Архитектура может быть аккуратно модульной, но multi-tenant-логика не входит в MVP.

## 1.2. Цели

- Реальное использование клубом.
- Демонстрация fullstack-разработки.
- Демонстрация AI-assisted development workflow.
- Демонстрация умения проектировать продукт, API, БД, тесты, деплой и документацию.

## 1.3. Портфолио-позиционирование

Проект должен быть оформлен как публичный инженерный кейс:

- публичный GitHub-репозиторий;
- README с описанием продукта;
- dev diary;
- prompt log;
- архитектурная документация;
- диаграмма БД;
- описание Codex workflow;
- видео-демо или GIF-сценарии;
- demo-режим без реальных персональных данных.

# 2. Пользовательские роли

## 2.1. Роли

В системе используются следующие роли:

1. **Гость**
2. **Зарегистрированный пользователь**
3. **Администратор клуба**
4. **Владелец клуба**

## 2.2. Гость

Гость может:

- просматривать главную страницу;
- смотреть расписание клуба;
- смотреть каталог игр;
- читать правила;
- смотреть контакты клуба;
- перейти к регистрации или входу.

Гость не может:

- создавать брони;
- создавать встречи;
- присоединяться к встречам;
- писать сообщения;
- видеть персональные данные пользователей.

## 2.3. Зарегистрированный пользователь

Пользователь может:

- управлять своим профилем;
- бронировать столы;
- просматривать свои брони;
- отменять свои брони в рамках правил клуба;
- создавать открытые игровые встречи;
- присоединяться к встречам;
- подтверждать участие во встречах;
- писать сообщения в ленте встречи;
- добавлять других пользователей в друзья;
- приглашать друзей во встречи;
- управлять настройками приватности;
- запросить выгрузку или удаление своих данных.

## 2.4. Администратор клуба

Администратор может:

- видеть заявки на бронирование;
- подтверждать брони;
- отменять брони;
- переносить брони;
- создавать брони вручную;
- видеть расписание клуба;
- управлять оперативными изменениями расписания, если это разрешено владельцем;
- получать уведомления о новых заявках;
- видеть минимально необходимые контакты пользователей;
- просматривать журнал действий в пределах своих полномочий.

Администратор не должен иметь доступ к избыточным персональным данным.

## 2.5. Владелец клуба

Владелец клуба может:

- управлять помещениями;
- управлять столами;
- настраивать вместимость столов;
- настраивать расписание клуба;
- настраивать исключения в расписании;
- блокировать помещения и столы;
- настраивать правила бронирования;
- управлять пользователями;
- блокировать пользователей;
- назначать и снимать администраторов;
- просматривать журнал действий;
- просматривать статистику;
- экспортировать данные в CSV/Excel;
- управлять юридическими документами и их версиями.

Приоритизация реализации: базовый `audit log` является обязательной частью MVP 1, потому что он нужен для безопасности, privacy и контроля действий с бронями и контактами. Статистика и CSV/Excel export входят в MVP 3 / portfolio polish, если клуб не потребует их раньше для реальной эксплуатации.

# 3. Персональные данные и доступ к контактам

## 3.1. Минимизация данных

Система должна собирать только те данные, которые нужны для работы клуба и связи с пользователями.

## 3.2. Данные пользователя

Поля профиля пользователя:

| Поле | Обязательность | Комментарий |
|---|---:|---|
| display name / имя или ник | обязательно | отображается другим пользователям |
| Telegram ID | обязательно при Telegram-входе | технический идентификатор |
| Telegram username | желательно | используется для связи, если доступен |
| телефон | обязателен для реального бронирования | для экстренной связи, так как Telegram может работать нестабильно; в БД поле должно быть nullable для удаления/анонимизации |
| email | опционально | альтернативный вход и восстановление доступа |
| настройки приватности | обязательно | определяют видимость контактов |
| согласия | обязательно | согласие на обработку ПДн, версия документов, дата, IP, user-agent |

## 3.3. Контакты, видимые администратору

Администратор видит только минимально необходимые контакты:

- display name;
- Telegram username, если доступен;
- телефон только при необходимости для экстренной связи;
- email по умолчанию не показывается администратору.

Доступ к полному телефону должен быть реализован как `break-glass` действие: в обычном интерфейсе телефон скрыт или маскирован, администратор явно нажимает действие для просмотра экстренного контакта, указывает причину при необходимости, а система обязательно пишет событие в `audit log`.

Технические идентификаторы, такие как Telegram ID, не должны отображаться в обычном интерфейсе администратора.

## 3.4. Контакты между пользователями

Пользователи могут видеть контакты друг друга только в рамках встреч и с учётом настроек приватности.

Рекомендуемая логика MVP:

- display name виден участникам встречи;
- Telegram username виден участникам встречи, если пользователь разрешил;
- телефон не показывается другим пользователям по умолчанию;
- email не показывается другим пользователям.

# 4. Бронирование столов

## 4.1. Общая логика

Клуб располагает несколькими помещениями. В помещениях находятся игровые столы.

Конфигурацию помещений и столов может изменять владелец клуба.

Все столы однотипны, но отличаются:

- условным номером;
- помещением;
- вместимостью;
- статусом доступности.

## 4.2. Помещения

Для помещения должны храниться:

- название;
- описание;
- активность;
- порядок отображения;
- ограничения доступности;
- список столов.

## 4.3. Столы

Для стола должны храниться:

- номер или название;
- помещение;
- вместимость;
- активность;
- возможность временной блокировки;
- комментарий для администратора или владельца.

## 4.4. Временные слоты

Бронирование осуществляется по фиксированным слотам с шагом **30 минут**.

Пользователь может:

- выбрать один слот;
- выбрать несколько последовательных слотов;
- забронировать стол на весь рабочий день клуба.

В БД бронь рекомендуется хранить как временной интервал:

```text
booking.start_at
booking.end_at
```

Слоты являются UI- и validation-слоем, а не отдельными сущностями брони.

## 4.5. Бронь на целый день

Бронь на целый день означает бронирование стола на весь рабочий день клуба в выбранную дату:

```text
start_at = время открытия клуба
end_at = время закрытия клуба
```

## 4.6. Подтверждение брони

Бронирование требует подтверждения администратором.

Пользователь создаёт заявку. Администратор подтверждает, переносит или отменяет её.

## 4.7. Статусы брони

Рекомендуемые финальные статусы:

| Статус | Значение |
|---|---|
| `pending` | заявка создана и ожидает подтверждения |
| `confirmed` | бронь подтверждена администратором |
| `cancelled_by_user` | бронь отменена пользователем |
| `cancelled_by_admin` | бронь отменена администратором |
| `completed` | бронь завершена |
| `expired` | заявка устарела или связанная встреча не состоялась |

## 4.8. История бронирований

Пользователь должен видеть историю своих бронирований:

- будущие брони;
- прошедшие брони;
- отменённые брони;
- статус каждой брони;
- связанные встречи, если есть.

## 4.9. Правила бронирования

Владелец клуба может настраивать:

- минимальное время до начала брони, после которого отмена запрещена;
- максимальное количество активных броней на пользователя;
- запрет бронирования для конкретного пользователя;
- блокировку помещения;
- блокировку конкретного стола;
- доступные дни и часы бронирования;
- ограничения на бронь на целый день.

## 4.10. Проверка конфликтов

Система должна предотвращать пересечение броней на один и тот же стол.

Критический сценарий:

> Два пользователя одновременно пытаются забронировать один и тот же стол на пересекающееся время. Система должна позволить создать или подтвердить только одну бронь.

Данный сценарий должен быть покрыт integration/concurrency-тестом.

# 5. Расписание клуба

## 5.1. Базовое расписание

Система должна поддерживать недельное расписание клуба:

- понедельник;
- вторник;
- среда;
- четверг;
- пятница;
- суббота;
- воскресенье.

Для каждого дня:

- открыт/закрыт;
- время открытия;
- время закрытия.

## 5.2. Исключения

Система должна поддерживать исключения по датам:

- праздничный день;
- выходной день;
- сокращённый день;
- продлённый день;
- особое расписание.

## 5.3. Блокировки ресурсов

Владелец или администратор, если ему разрешено, может временно заблокировать:

- помещение;
- конкретный стол.

Причины блокировки:

- ремонт;
- мероприятие;
- техническая причина;
- внутреннее использование клуба.

# 6. Игровые встречи и поиск игроков

## 6.1. Назначение

Функция игровых встреч позволяет пользователям искать других игроков под конкретную настольную игру или варгейм.

Пример сценария:

> Пользователь создаёт встречу: «Ищу 3 игроков в Dune Imperium в субботу в 18:00».

## 6.2. Каталог игр

Система должна иметь каталог игр клуба.

Для игры хранятся:

- название;
- жанр;
- примерная длительность;
- рекомендуемое количество игроков;
- сложность;
- описание;
- фото;
- активность;
- теги.

## 6.3. Своя игра

Пользователь может создать встречу:

- по игре из каталога;
- по своей игре, указав название текстом.

Для этого в модели встречи должны поддерживаться поля:

```text
game_id nullable
custom_game_title nullable
```

## 6.4. Создание встречи

При создании встречи пользователь указывает:

- игру из каталога или своё название;
- дату;
- время;
- желаемую длительность;
- нужное количество игроков;
- описание;
- теги;
- подходит ли новичкам;
- нужен ли ведущий;
- дополнительные условия.

## 6.5. Резервирование стола

Встреча **не резервирует стол сразу**.

Финальное решение:

> Встреча сначала собирает участников. После набора нужного количества игроков создаётся заявка на бронирование стола. Затем администратор подтверждает бронь.

Если нужное количество игроков не набралось, бронирование не происходит.

## 6.6. Статусы встречи

Рекомендуемые статусы встречи:

| Статус | Значение |
|---|---|
| `open` | встреча открыта для участников |
| `ready_to_book` | нужное количество участников набрано |
| `booking_pending` | заявка на бронь создана и ждёт подтверждения |
| `confirmed` | бронь подтверждена |
| `cancelled` | встреча отменена |
| `completed` | встреча завершена |
| `expired` | встреча не набрала участников или время прошло |

## 6.7. Участники встречи

Участники могут:

- присоединиться к встрече;
- выйти из встречи;
- подтвердить участие;
- видеть список участников;
- видеть разрешённые контакты участников;
- писать сообщения в ленту встречи.

## 6.8. Теги встреч

Встречи должны поддерживать теги, например:

- новичкам подходит;
- нужен ведущий;
- игра на 2000 очков;
- своя игра;
- игра клуба;
- обучение;
- турнирный формат.

## 6.9. Лента сообщений

В MVP чат реализуется как **простая лента сообщений**, а не полноценный realtime-чат.

Особенности MVP:

- сообщения видны только участникам встречи;
- сообщения сохраняются в БД;
- обновление может происходить через refresh или polling;
- WebSocket/SSE не обязательны в MVP;
- модерация может быть минимальной.

Realtime-чат может быть добавлен позже.

## 6.10. Друзья

В MVP поддерживается минимальная логика друзей:

- добавить пользователя в друзья;
- список друзей;
- пригласить друга во встречу.

Не входят в MVP:

- социальная лента;
- приватные сообщения;
- рекомендации друзей;
- сложные настройки видимости.

# 7. Регистрация и авторизация

## 7.1. Основной подход

Основной канал авторизации — Telegram.

Telegram используется:

- как способ входа;
- как канал уведомлений;
- как альтернатива сайту;
- как основа для Telegram Mini App.

## 7.2. Telegram-вход

Система должна поддерживать:

- вход через Telegram на сайте;
- связку аккаунта через Telegram-бота;
- авторизацию Telegram Mini App через Telegram init data.

## 7.3. Email

Email является опциональным.

Назначение email:

- альтернативный вход;
- восстановление доступа;
- дополнительный канал связи, если пользователь указал его сам.

## 7.4. Пароли

Парольная регистрация не входит в MVP.

Рекомендуемый подход:

- Telegram-first;
- email magic link или одноразовый код как альтернативный способ входа в будущем.

## 7.5. Телефон

Телефон собирается обязательно для экстренной связи, так как Telegram может работать нестабильно.

Требования к телефону:

- хранить только необходимое значение;
- не показывать другим пользователям;
- показывать администратору только при необходимости;
- не использовать как основной способ входа в MVP;
- не подключать SMS OTP в MVP.

# 8. Telegram-бот и Telegram Mini App

## 8.1. Назначение бота

Telegram-бот должен:

- отправлять уведомления пользователям;
- отправлять уведомления администраторам;
- позволять бронировать столы;
- позволять просматривать свои брони;
- позволять отменять бронь;
- позволять подтверждать участие во встрече;
- открывать Telegram Mini App;
- работать как альтернатива сайту для основных пользовательских сценариев.

## 8.2. Telegram Mini App

Telegram Mini App входит в целевую архитектуру.

Финальное решение:

> Telegram Mini App будет реализован, но архитектуру нужно готовить сразу. На первом этапе допустимо реализовать web app и bot notifications, затем переиспользовать frontend-компоненты в Mini App.

## 8.3. Рекомендуемая роль каналов

| Канал | Назначение |
|---|---|
| Сайт | публичный лендинг и web-приложение |
| Telegram Bot | уведомления, быстрые действия, запуск Mini App |
| Telegram Mini App | бронирование и встречи внутри Telegram |
| Админка | web-only интерфейс для администраторов и владельца |

## 8.4. Команды бота

Пользовательские команды:

```text
/start — регистрация или связка аккаунта
/help — помощь
/book — открыть бронирование или Mini App
/my_bookings — мои брони
/my_meetups — мои встречи
/find_game — найти открытые встречи
/cancel — отменить бронь или участие
/settings — настройки уведомлений и приватности
```

Администраторские команды:

```text
/admin — админское меню
/pending — заявки на подтверждение
/today — брони на сегодня
```

Основной UX должен строиться не только на командах, но и на кнопках, inline-кнопках и переходах в Mini App.

## 8.5. Уведомления

Бот должен уведомлять:

- пользователя о создании заявки;
- пользователя о подтверждении брони;
- пользователя об отмене или переносе брони;
- пользователя о напоминании перед игрой;
- участников о статусе встречи;
- админа о новых заявках;
- админа о важных изменениях.

## 8.6. Telegram update mode

Для разработки:

- polling.

Для staging и production:

- webhook.

# 9. Сайт и фронтенд

## 9.1. Назначение сайта

Сайт должен совмещать:

- публичный лендинг клуба;
- frontend-приложение для бронирования;
- личный кабинет пользователя.

## 9.2. Публичные страницы

Публичные страницы:

- главная;
- расписание;
- игры;
- правила;
- контакты.

## 9.3. Приватные страницы

# 10. Админская панель

## 10.1. Назначение

Админская панель предназначена для администраторов и владельца клуба.

## 10.2. Функции администратора

Администратор может:

- просматривать заявки на бронирование;
- подтверждать брони;
- переносить брони;
- отменять брони;
- создавать брони вручную;
- просматривать брони на день;
- видеть минимально необходимые контакты пользователей;
- получать уведомления о новых заявках.

## 10.3. Функции владельца

Владелец может:

- управлять помещениями;
- управлять столами;
- управлять расписанием клуба;
- управлять исключениями расписания;
- блокировать помещения;
- блокировать столы;
- управлять пользователями;
- блокировать пользователей;
- назначать администраторов;
- настраивать правила бронирования;
- просматривать журнал действий;
- просматривать статистику;
- экспортировать данные.

Приоритет реализации: журнал действий обязателен для MVP 1. Статистика и экспорт являются owner-функциями, но реализуются в MVP 3 / portfolio polish, если клуб не потребует их раньше.

## 10.4. Статистика

Статистика нужна владельцу клуба, но не блокирует MVP 1. По умолчанию она относится к MVP 3 / portfolio polish, если клуб не потребует её раньше.

Админка должна поддерживать статистику:

- загрузка столов;
- популярные игры;
- количество броней;
- количество отмен;
- активность встреч;
- динамика бронирований по дням.

## 10.5. Экспорт

CSV/Excel export нужен владельцу клуба, но не блокирует MVP 1. По умолчанию он относится к MVP 3 / portfolio polish. Любой экспорт персональных данных должен писаться в `audit log`.

Нужен экспорт в:

- CSV;
- Excel.

## 10.6. Журнал действий

Базовый `audit log` является обязательной частью MVP 1. Он нужен для контроля действий администраторов и владельца, расследования спорных ситуаций, доступа к экстренным контактам и privacy/compliance.

Система должна вести audit log:

- кто выполнил действие;
- когда;
- какое действие;
- над какой сущностью;
- старое и новое значение, если применимо;
- IP/user-agent, если применимо и допустимо.

# 13. Технологический стек

## 13.1. Общий подход

Рекомендуемый стек — TypeScript-first monorepo.

## 13.2. Рекомендуемый стек

```text
Monorepo: pnpm workspaces или Turborepo
Frontend: Next.js
Backend: NestJS
Bot: grammY или Telegraf
Database: PostgreSQL
ORM: Prisma
Cache/queues: Redis + BullMQ
UI: Tailwind CSS + shadcn/ui
Testing: Vitest/Jest, Supertest, Playwright
API docs: Swagger/OpenAPI
Deploy: Docker Compose на VPS
```

## 13.3. Причины выбора TypeScript

- один язык для frontend, backend, bot и shared packages;
- удобно для monorepo;
- хорошо подходит для Codex-задач;
- можно переиспользовать типы и схемы;
- Next.js подходит для сайта и web app;
- NestJS подходит для структурированного backend API;
- Telegram-бот хорошо интегрируется в Node.js-экосистему.

## 13.4. PostgreSQL

Основная БД — PostgreSQL.

PostgreSQL используется для:

- пользователей;
- ролей;
- помещений;
- столов;
- расписания;
- броней;
- встреч;
- сообщений;
- друзей;
- каталога игр;
- audit log;
- consent log.

## 13.5. Redis

Redis нужен для:

- очередей уведомлений;
- напоминаний перед игрой;
- rate limiting;
- временных задач;
- обработки Telegram webhook;
- возможных hold-резерваций в будущем.

Redis желательно заложить в архитектуру сразу.

## 13.6. Docker

Docker и docker-compose нужны с первого этапа.

Минимальные сервисы:

```text
postgres
redis
backend
frontend
bot
```

Для локальной разработки допустимо запускать backend/frontend/bot вне контейнера, но PostgreSQL и Redis — через Docker.

Рекомендуемая структура:

# 14. Тестирование и качество

## 14.1. Требуемые уровни тестирования

Для портфолио и надёжности проекта нужны:

| Уровень | Назначение |
|---|---|
| Unit tests | бизнес-правила и доменная логика |
| Integration tests | API + PostgreSQL + сервисы |
| API e2e tests | полные сценарии через HTTP API |
| UI e2e tests | ключевые пользовательские сценарии через браузер |
| Bot tests | команды, callback-кнопки, уведомления |
| Concurrency tests | защита от двойного бронирования |
| Authorization tests | проверка прав доступа |

## 14.2. Критические сценарии

Обязательно покрыть:

1. Пользователь создаёт заявку на бронь.
2. Админ подтверждает бронь.
3. Пользователь отменяет бронь в допустимое время.
4. Пользователь не может отменить бронь слишком поздно.
5. Заблокированный пользователь не может бронировать.
6. Два пользователя не могут забронировать один стол на пересекающееся время.
7. Пользователь не может видеть чужие персональные данные.
8. Встреча создаётся без брони.
9. После набора участников создаётся заявка на бронь.
10. Если участников не набралось, бронь не создаётся.

Seed должен включать:

# 16. Юридические и privacy-требования

## 16.1. Оператор персональных данных

Оператором персональных данных является клуб.

## 16.2. Необходимые документы

Нужны:

- политика конфиденциальности;
- пользовательское соглашение;
- согласие на обработку персональных данных;
- правила клуба;
- порядок удаления аккаунта;
- порядок выгрузки данных.

Юридические тексты должны быть проверены ответственным лицом или юристом клуба.

## 16.3. Согласия

Система должна хранить факт согласия:

- user id;
- тип согласия;
- версия документа;
- дата и время;
- IP;
- user-agent.

## 16.4. Удаление аккаунта

Пользователь должен иметь возможность запросить удаление аккаунта.

Рекомендуемая логика:

- пользователь отправляет запрос;
- система фиксирует запрос;
- администратор или владелец обрабатывает его;
- персональные данные удаляются или анонимизируются;
- исторические брони могут сохраняться в обезличенном виде для статистики и аудита.

## 16.5. Выгрузка данных

Пользователь должен иметь возможность запросить выгрузку своих данных.

Формат:

- JSON;
- возможно CSV для бронирований.

## 16.6. Cookies и аналитика

Финальное решение для MVP:

- использовать только технические cookies;
- не использовать рекламные пиксели;
- не использовать ретаргетинг;
- не использовать запись пользовательских сессий;
- не использовать внешнюю поведенческую аналитику в production MVP.

Допустимы:

- технические cookies для auth/session;
- серверная агрегированная статистика внутри админки;
- self-hosted privacy-friendly analytics позже, если потребуется.
