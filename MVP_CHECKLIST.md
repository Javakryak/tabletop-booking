# MVP_CHECKLIST.md

## Назначение документа

Этот документ фиксирует критерии готовности MVP приложения для бронирования игровых столов клуба настольных игр и варгеймов.

## Уточнение по audit / телефону / export

- Базовый `audit log` является обязательной частью MVP 1.
- Полный телефон пользователя хранится для экстренной связи: поле nullable в БД, но бизнес-правило требует заполненный телефон перед созданием реальной брони.
- Администратор видит телефон только в маскированном виде по умолчанию. Полный телефон доступен только через явное emergency/break-glass действие, которое обязательно пишется в `audit log`.
- Статистика и CSV/Excel export относятся к MVP 3 / portfolio polish, если клуб не потребует их раньше.

---

MVP считается готовым не тогда, когда реализованы все идеи проекта, а когда приложение можно безопасно показать клубу, использовать в ограниченном production-режиме и демонстрировать как портфолио-проект AI-assisted developer.

## Статусы проверки

Используйте чекбоксы при ревью MVP:

- `[ ]` не начато;
- `[~]` частично готово;
- `[x]` готово;
- `[!]` требует решения перед production.

# 1. Product Scope

## 1.1. Общие критерии

- [ ] Приложение рассчитано на один конкретный клуб.
- [ ] В проекте нет логики multi-club/SaaS, если она не требуется MVP.
- [ ] Платные услуги и онлайн-оплаты не реализуются.

# 5. Booking Flow

## 5.1. Booking creation

- [ ] Пользователь может выбрать дату.
- [ ] Пользователь видит доступные интервалы с шагом 30 минут.
- [ ] Пользователь может выбрать несколько слотов подряд.
- [ ] Пользователь может выбрать бронирование на весь рабочий день.
- [ ] Бронирование на весь день хранится как обычный интервал `start_at` / `end_at`.
- [ ] Пользователь может выбрать подходящий стол.
- [ ] Система проверяет расписание клуба.
- [ ] Система проверяет блокировки помещения и стола.
- [ ] Система проверяет лимит активных броней пользователя.
- [ ] После создания бронь получает статус `pending`.
- [ ] Пользователь получает уведомление о создании заявки.
- [ ] Администратор получает уведомление о новой заявке.

## 5.2. Booking statuses

- [ ] Поддерживается статус `pending`.
- [ ] Поддерживается статус `confirmed`.
- [ ] Поддерживается статус `cancelled_by_user`.
- [ ] Поддерживается статус `cancelled_by_admin`.
- [ ] Поддерживается статус `completed`.
- [ ] Поддерживается статус `expired`, если используется.
- [ ] История изменения статусов сохраняется.

## 5.3. Admin confirmation

- [ ] Администратор видит список заявок `pending`.
- [ ] Администратор может подтвердить заявку.
- [ ] При подтверждении система повторно проверяет отсутствие конфликта.
- [ ] При подтверждении бронь получает статус `confirmed`.
- [ ] Пользователь получает уведомление о подтверждении.
- [ ] Действие администратора попадает в audit log.

## 5.4. Cancellation

- [ ] Пользователь может отменить свою бронь.
- [ ] Система проверяет минимальное время до отмены.
- [ ] Если отмена запрещена правилами, пользователь получает понятную ошибку.
- [ ] Администратор может отменить бронь.
- [ ] Владелец может отменить бронь.
- [ ] При отмене пользователь получает уведомление.
- [ ] Отмена сохраняется в истории статусов.

## 5.5. Booking history

- [ ] Пользователь видит свои прошлые брони.
- [ ] Пользователь видит свои будущие брони.
- [ ] Пользователь не видит чужие персональные данные.
- [ ] Администратор видит клубное расписание без лишних данных.

# 7. Concurrency and Data Integrity

## 7.1. Critical race condition

- [ ] Реализована защита от двойного бронирования одного стола на пересекающееся время.
- [ ] Проверка конфликта происходит внутри транзакции.
- [ ] Пересечение интервалов проверяется по правилу `A.start_at < B.end_at AND A.end_at > B.start_at`.
- [ ] Учитываются только активные статусы брони.
- [ ] Есть integration test на одновременное бронирование.
- [ ] В тесте только одна из двух конкурентных заявок может быть успешно подтверждена.

## 7.2. Database constraints

- [ ] Есть внешние ключи для основных связей.
- [ ] Есть индексы для частых запросов.
- [ ] Есть индексы для поиска броней по дате, столу и статусу.
- [ ] Есть индексы для Telegram ID.
- [ ] Есть индексы для audit log.
- [ ] Soft delete не ломает уникальные ограничения.

- [ ] `.env` не попадает в git.
- [ ] `.env.example` не содержит реальных секретов.

- [ ] Telegram bot token не логируется.

- [ ] Ownership проверяется на backend.
